# 剪贴板助手 — 项目总结报告

## 项目目标
解决 Claude Code（Windows 终端）无法直接粘贴截图的问题。截图后自动保存到桌面 `~/Desktop/temp/`，路径自动复制到系统剪贴板，用户 Ctrl+V 即可粘贴路径到对话窗口。

---

## 技术方案

### 架构

```
┌─────────────┐     ┌──────────────┐     ┌─────────────────┐
│ Claude Code │──▶  │  start-hook  │──▶  │ clipboard-monitor│
│  SessionStart│     │  (PID锁检查) │     │  (后台轮询进程)   │
│  Stop        │◀──  │  stop-hook   │◀──  │  (PID 锁单例)    │
└─────────────┘     └──────────────┘     └────────┬─────────┘
                                                  │
                                    ┌─────────────▼──────────┐
                                    │ ~/Desktop/temp/        │
                                    │ screenshot_YYYYMMDD_   │
                                    │    HHMMSS.png          │
                                    │ (路径自动复制到剪贴板)   │
                                    └────────────────────────┘
```

### 核心脚本（`C:\Users\PC\.claude\scripts\`）

| 文件 | 作用 |
|------|------|
| `clipboard-monitor.py` | 后台循环监听剪贴板，每 500ms 轮询，检测到新截图自动保存并复制路径 |
| `start-hook.py` | SessionStart 钩子调用，检查 PID 锁，未启动则启动监控进程 |
| `stop-hook.py` | Stop 钩子调用，根据 PID 锁直接终止监控进程 |

### Hooks 配置（`settings.json`）

- **SessionStart** → `start-hook.py` — CC 启动时自动触发
- **Stop** → `stop-hook.py` + 完成提示音 — CC 关闭时自动触发

## 迭代历程

### 第 1 轮：方案探索
- 调研 Claude Code 图片粘贴机制：Alt+V 在 Windows 上有 bug，Ctrl+V 仅支持文本
- 尝试 slash command + keybinding 方案（Alt+Z 触发），但响应慢需等 AI 处理
- 尝试 bash bind 方案，但 Claude Code TUI 不响应 bash 行编辑绑定
- **最终采用后台监听方案**：Python 脚本每 500ms 轮询剪贴板

### 第 2 轮：自动启停配置
- 创建 `start-hook.py` / `stop-hook.py`
- 配置 `settings.json` 的 `SessionStart` / `Stop` hooks
- 切换模型时确认 hooks 配置不会被覆盖

### 第 3 轮：修复重复启动问题
- **问题**：多 Claude Code 窗口各自触发 Startup hook，多个监听实例同时运行，桌面生成多个重复截图
- **解决**：引入文件独占锁（PID 锁），多窗口竞争时只有第一个实例能获取锁
- 截图保存目录从 `~/Desktop` 迁移到 `~/Desktop/temp`

### 第 4 轮：计数器方案升级
- **问题**：PID 锁在 Windows 下不够可靠
- **解决**：改用计数器方案（`.cc-monitor-counter`），多窗口协调启停
- 第一个窗口启动监控，最后一个窗口关闭时停止

### 第 5 轮：修复 PowerShell 弹窗
- **问题**：`powershell.exe Set-Clipboard` 保存路径时会闪弹窗
- **解决**：改用 `win32clipboard` 直接在 Python 内操作剪贴板，无弹窗

### 第 6 轮：清理
- 删除 11 个旧版本/废弃脚本
- 仅保留 3 个核心脚本 + 运行时文件

### 第 7 轮：简化启停机制
- **问题**：计数器方案过于复杂，Stop hook 直接杀进程更简单
- **解决**：废弃计数器，`stop-hook.py` 直接 `taskkill` 杀进程
- 文件名从 `start-monitor.py` / `stop-monitor.py` 改为 `start-hook.py` / `stop-hook.py`

### 第 8 轮：修复 Monitor 进程崩溃
- **问题**：保存一张截图后 monitor 进程退出，后续截图不保存
- **原因**：保存后剪贴板被替换为文件路径文本，`ImageGrab.grabclipboard()` 返回 `DibImageFile`，非标准 `Image.Image`，导致异常
- **解决**：用 `isinstance(img, Image.Image)` 严格判断剪贴板内容类型

---

## 关键技术点

| 问题 | 解决方案 |
|------|---------|
| 进程单例 | PID 锁文件 `.cc-monitor.lock` |
| 文件名冲突 | 时间戳命名 `screenshot_YYYYMMDD_HHMMSS.png` |
| PowerShell 弹窗 | `win32clipboard` 替代 `Set-Clipboard` |
| 后台进程无感 | `pythonw.exe` + `DETACHED_PROCESS` 模式 |
| Monitor 崩溃退出 | `isinstance(img, Image.Image)` 严格判断剪贴板类型 |
| 资源消耗 | CPU <0.1%，内存 ~30MB |
